## TL;DR (read first)

**The honest answer to "4500 tok/s on Qwen3.6-27B on this 4090":** that target is **not achievable on this specific hardware setup tonight**, because of a fundamental memory-budget constraint, not a software bug.

| Slot | What |
|---|---|
| 4090 total VRAM | 23.51 GiB |
| Untouchable resident processes (display, prod, colleague, your uvicorn backend on port 8110) | ~3.3 GiB |
| Free for the model + KV + activations + workspace | ~20.2 GiB |
| Qwen3.6-27B AWQ-INT4 weights alone | ~18.0 GiB |
| Left over for KV cache + activations + FlashInfer prefill workspace | ~2.2 GiB |

**At 2.2 GiB residual, the math doesn't support the stated UX goal:**
- 16 concurrent agents × 8K context with FP8 KV needs **~8.5 GiB** of KV alone.
- 1 agent × 32K context needs ~2.1 GiB.
- The "frontier-API-grade local inference" experience requires *headroom*; we have *deficit*.

**What does work today (the recipe is committed and reproducible):**
- vLLM 0.19.1 + AWQ-Int4 weights, **single-stream**, **max-model-len ≤ 1024 tokens**, **max-num-seqs=1**, FlashInfer workspace shrunk to 128 MiB, `--enforce-eager`.
- ~21 tok/s decode, ~160 ms TTFT on small prompts (256 tokens in / 64–256 out).
- Coherent thinking-mode output validated end-to-end via `/v1/chat/completions`.

**What unblocks the user's goal:**
1. **Free 3 GiB of VRAM** by stopping the uvicorn `backend.main:app` on port 8110 (PID 1658309 holds 1.86 GiB) and any other GPU residents you control. With +3 GiB headroom, max-model-len ≥ 4K and max-num-seqs ≥ 4 become viable.
2. **Use a 48 GB GPU** (RTX A6000 / RTX 6000 Ada) — Qwen3.6-27B + concurrent serving is comfortable there.
3. **Fix the TurboQuantDC vLLM integration** (extensive code review surfaced the work needed). Once 3-bit KV is wired into vLLM, concurrent capacity at 8K context goes ~5× — but the integration as it stands tonight (`turboquantdc/vllm_integration.py`) has 7 critical bugs and is not production-viable. See the code review for the patch list.
4. **Use a smaller Qwen3.6 variant** for this hardware (you ruled this out: "no other model").

The autonomous run produced: a working vLLM serve recipe; a benchmark harness, a NIAH validator, a sweep runner; a comprehensive code-review synthesis from 8 parallel Opus 4.7 subagents; honest numbers; and a clear path forward.

## Model

**Pinned:** `cyankiwi/Qwen3.6-27B-AWQ-INT4` (HF Hub) → local at `models/Qwen3.6-27B-AWQ-INT4/`.

- 19 GB on disk, 18.0 GiB on GPU after load.
- Architecture: `Qwen3_5ForConditionalGeneration` (multimodal — vision encoder skipped via `--language-model-only`).
- Quantization format: `compressed-tensors` (pack-quantized, num_bits=4, group_size=32).
- Native context: 262K (we cap to 1K–2K due to VRAM).
- Hybrid attention: 3:1 ratio of GDN (gated delta network linear-attention layers) to standard attention. vLLM treats GDN layers as Mamba and uses a 1568-token attention block to align mamba page sizes.

The official `Qwen/Qwen3.6-27B-FP8` (28 GB) does not fit on this GPU. Other variants (Q3_K_M, Q4_K_S GGUFs) were not tried tonight; would also fit but with throughput cost vs Marlin AWQ.

## TurboQuantDC integration status

**Not shipped tonight.** Per the parallel code-review run (8 Opus 4.7 subagents on the existing repo), `turboquantdc/vllm_integration.py` has **7 CRITICAL bugs** that prevent correct or efficient operation in vLLM:

1. `TurboQuantAttentionBackend` is stateless — `compress_kv` never appends to its store; autoregressive decoding throws away history every step.
2. GQA is broken — queries flattened across all heads attend to keys from all KV heads, not their group. Hard blocker for Qwen3.6-27B (4 KV heads, GQA factor 12).
3. fp16 underflow in `estimator.py` — `1e-8` epsilon is below fp16 min normal; division silently produces inf/nan.
4. PagedAttention layout incompatible — `TurboQuantCacheManager` uses flat `(max_seq_len, H, D)` instead of vLLM's `(num_blocks, block_size, num_kv_heads, head_dim)`.
5. Mean-removal not wired in — the very fix that takes Qwen PPL from 9410 → 7.90 lives in `_CompressedLayer`, not in `vllm_integration.py`. Demo through this path would PPL-explode.
6. The "monkey-patch example" targets a vLLM internal API (`llm.llm_engine.model_executor...`) that was removed in V1 engine refactor (default since vLLM 0.8). vLLM 0.19.1 does not expose this layout.
7. `int64` index blowup — `mse_indices.long()` is 8× the int16 stored on disk.

**Plus** an algorithm-review finding: `e8_lattice.py:nearest_e8_relaxed` (the function that produced the "E8 lattice VQ" results in HANDOFF.md) is mathematically **half-integer scalar quantization**, not E8. The Conway-Sloane / Viazovska / QuIP# E8P12 references should be retracted from the file. The technique still works, but is mislabeled.

The full review with line-level fixes is at `docs/code_review/2026-04-27/CODE_REVIEW_2026-04-27.md`. Per-area reports at `docs/code_review/2026-04-27/0[1-8]_*.md`.
