# Hot-Path Correctness Review (Opus 4.7)

Scope: `vllm_integration.py`, `kv_cache.py`, `estimator.py`, `polarquant.py`, `qjl.py` (read-only).
Reviewer mindset: this code runs inside vLLM's attention loop on every token. Anything that
mutates shared state, allocates, or breaks shape contracts under GQA / batch>1 / paged blocks
is a ship-stopper.

---

## CRITICAL  (bugs that produce wrong output / crash / memory unsafety / silently degrade quality)

- `turboquantdc/rotation.py:122-137` (called from every quantize/dequantize) — **`fast_wht` mutates its input in place via `.view()` aliased writes**. The function takes `x`, builds `xe = x.view(...)`, then assigns `xe[..., 0, :] = s` which writes through the alias back into `x`. This means `apply_wht_rotation` first does `result = x.clone()` (good) — but during decode, `polarquant.rotate()` and `polarquant.unrotate()` call `apply_wht_rotation` directly on freshly-passed tensors. Inside `unrotate`, `result = fast_wht(result)` (line 210) returns the same tensor it was given; then `result = result * signs` allocates a new tensor. So far OK because of the clone. **However**, the in-place write inside `fast_wht` will fail or silently corrupt if the input tensor is non-contiguous (`.view` will throw) or is a non-leaf tensor that requires grad. Under vLLM, queries arrive as slices/transposes of larger buffers (very often non-contiguous after `view(B, H, T, D).transpose`), so `apply_wht_rotation(query, ...)` will raise `RuntimeError: view size is not compatible with input tensor's size and stride` the first time a real query hits it. **Fix:** in `apply_wht_rotation` line 202 keep `result = x.clone().contiguous()`, and in `fast_wht` use `reshape` (or assert contiguous and document); also `fast_wht` should not mutate — return a new tensor. Since this is not in your review file list, flagged because it is the actual hot-path call site for `polarquant.rotate`.

- `turboquantdc/polarquant.py:144` — **`PolarQuant.quantize` does not normalize the input.** Both `quantize` and `dequantize` operate on whatever vector you give them. The codebook centroids were solved for the post-rotation distribution of a *unit* vector (Lemma 1 → Beta/Gaussian with variance `1/d`). If a key with norm 8 is passed in (typical for Qwen3 keys after RoPE), the rotated coordinates will have variance `64/d`, but Lloyd-Max only knows centroids in `[-3.5/sqrt(d), +3.5/sqrt(d)]`. Every coordinate gets clipped to one of the two outer centroids. The class docstring says "for non-unit vectors, the caller is responsible for normalizing" — but `TurboQuantEstimator.quantize` does normalize, while `TurboQuantKVCache.append` (line 100) and `TurboQuantAttentionBackend.compress_kv` (line 255) call `value_quantizer.quantize(values_normalized)` correctly, but `PolarQuant.quantize` is *also* called transitively from `TurboQuantEstimator.polar.quantize(x_normalized)` (estimator.py:102, OK). The **bug** is: `PolarQuant.forward` (polarquant.py:165-176, used by tests and any direct user) does not normalize either, so any unit-test of `PolarQuant` that uses non-unit vectors will silently produce garbage and you won't see it because the codebook saturates. Not strictly a hot-path bug, but the contract is fragile. **Fix:** either (a) raise if `||x||` deviates from 1 by > 5%, or (b) normalize internally and return both `indices` and `vec_norm`. Document the precondition in the type signature (e.g. rename the param to `x_unit`).

- `turboquantdc/estimator.py:99,105` — **fp16 underflow in `vec_norm` and `residual` for long-context Qwen3.** Inputs `x` arrive as fp16 from vLLM. `x.norm(dim=-1)` in fp16 with d=128 sums 128 squared values; if the per-coordinate magnitude is ~1e-2 (typical post-LayerNorm key entries), the squared sum is ~1.3e-2 — fine. But for value vectors after deep-layer attention output, individual entries can be 1e-3 with occasional 1e-1 outliers; squaring 1e-3 gives 1e-6, sum-of-128 still representable, but the divide `x / (vec_norm + 1e-8)` (estimator.py:99) at fp16 will produce values up to ~1e5 if `vec_norm` underflows to ~1e-5 — the `1e-8` floor is **way below fp16 min normal (6e-5)**, so it does nothing in fp16. The result is `inf` after the divide, then `nan` after `polar.quantize`. **Fix:** cast to fp32 for the norm/normalize/quantize chain (this code already implicitly does so via `torch.norm` → fp32 accumulator on most paths, but the final division is back in fp16). Explicitly: `x32 = x.float(); vec_norm = x32.norm(...).clamp_min(1e-6); x_normalized = x32 / vec_norm`. This is essential at 8K+ context where outlier value vectors are common.

- `turboquantdc/vllm_integration.py:586-587` — **`store()` upcasts to fp32 inside the hot-path** via `keys.reshape(N * H, D).float()`. With `(N=1, H=8, D=128)` per decode token across 64 layers, that's 64 fp32 conversions of (1024-element) tensors per token — bearable. But `compute_attention` does **not** do the same upcast for `queries_flat` before passing to `inner_product`. Inside `TurboQuantEstimator.inner_product` (estimator.py:171), `polar.dequantize(mse_indices)` returns fp32 (centroids are registered fp32), but `query @ x_mse.T` will then be fp32-dtype matmul because PyTorch promotes. So far so good. **The actual bug:** `TurboQuantCacheManager.fetch` (vllm_integration.py:670-676) returns `signs_float`, `rnorm.float()`, `vnorm.float()` — all fp32 — but the centroid lookup in `polar.dequantize(mse_indices.long())` uses the *layer's* PolarQuant centroids buffer which (at line 94 of polarquant.py) is `to(device)` of whatever the user passed. If the user constructs `TurboQuantCacheManager(device="cuda")`, the centroids are float32 on cuda. But the returned `compressed_keys["mse_indices"]` is `.long()` (int64) — that's a **8x memory blowup vs the int16 stored on disk**. Per fetched token, you allocate an int64 (N*H*D*8) bytes on the GPU. At seq_len=32768 and D=128 H=8, that's 268 MB allocated *per fetch* just to satisfy `centroids[indices]` indexing. **Fix:** PyTorch supports `int32` and `int16` indexing on CUDA in 2.x — keep `mse_indices` as int16 and pass directly; or pre-cast the centroids buffer once and use `torch.gather` on int16. At minimum, use int32, not int64.

- `turboquantdc/vllm_integration.py:622-683` — **`fetch()` is incompatible with vLLM PagedAttention block layout.** vLLM stores KV in physical blocks of shape `(num_blocks, block_size, num_kv_heads, head_dim)` and a `block_table` mapping logical sequence positions → block IDs. `fetch(layer_idx, slot_indices)` does `self._key_mse_buf[layer_idx][slot_indices]` which assumes a flat `(max_seq_len, H, D)` layout — i.e. one contiguous KV cache per layer per **request**. This will (a) break under continuous batching where multiple sequences share the buffer pool; (b) require copying compressed data on every preemption/swap; (c) silently use slot N from request A as slot N for request B. The integration as-is supports a single request at a time, with no block awareness. The 16-32 concurrent stream goal **cannot be met with this code path**. **Fix:** either (i) restructure buffers as `(num_blocks, block_size, num_kv_heads, ...)` and add a `block_tables`-aware `fetch`, or (ii) abandon the `TurboQuantCacheManager` for vLLM and only use the `monkey-patch backend.compute_attention` flow (the docstring already calls this the "fastest path"). The cache manager looks like dead code for the vLLM target.

- `turboquantdc/vllm_integration.py:127-199` — **`TurboQuantAttentionBackend` holds per-layer mutable state in `self._compressed_key_store` / `_compressed_value_store` lists**, but `compress_kv` (line 205) **never appends to them**. They are allocated empty in `__init__` and only read by `memory_usage`. This means `memory_usage` always reports zero, *and* worse — if anyone follows the docstring at line 50 (`backend.compress_kv(...)` then `backend.compute_attention(...)`), there is **no shared state for the next decode step**. Each decode call re-compresses the entire context from scratch, throwing away all previous compressed KV. This will either (a) silently produce a 200x slowdown on long contexts because all keys are recomputed each step, or (b) crash because `compute_attention` only sees the current token's K/V, not the past. The "monkey-patch path" example at lines 44-54 has the same bug: `compressed = backend.compress_kv(lidx, key, value)` followed by `compute_attention(..., *compressed, ...)` only ever sees the **current step's K/V**, not the cache. **Fix:** either (a) make `compress_kv` append to the per-layer store and have `compute_attention` gather across the store, or (b) document that this class is stateless and require the caller to pass a `(prev_compressed, new_kv)` tuple. The current state cannot work for autoregressive decoding.

- `turboquantdc/qjl.py:78` — **Sign quantization rule `(projected >= 0).float() * 2.0 - 1.0` maps zeros to +1, biasing the estimator on zero-residual coordinates.** For 4-bit MSE keys the residual can have exactly-zero entries after rotation+dequant in fp32 (rare but happens). More importantly, on **prefill** the very first token's key has zero norm after the residual is computed (since the first token has no context to attend to in some encoders, but this isn't the issue here) — the real issue: `S @ r` for `r = 0` (which can happen if a key is exactly representable by the codebook) gives all-zero `projected`, all signs become +1, and `correction = ||r|| * stuff = 0 * stuff = 0` — so it is fortunately benign for the **zero residual** case because `||r||` zeros it out. **However**, partial zeros in `S @ r` (from the linear combination summing exactly to 0 in fp32, which is plausible at d=128 with fp16-input-projection) bias the QJL inner product upward. The bias is small (probability ~1/2^32 per coordinate) but **it stacks**: `m=128` projections per key, K=32768 keys, that's potentially millions of zero-points. **Fix:** use `torch.sign(projected) + (projected == 0).float()` or sample a tie-break per coordinate at init time. Marginal in practice but a known QJL pitfall.

## HIGH  (production blockers / clear correctness risk under load)

- `turboquantdc/qjl.py:103` — **QJL projection scale is wrong when `m != d`.** The paper's correction is `sqrt(pi/2) / sqrt(m)`, not `sqrt(pi/2) / m`. The docstring at line 13 even spells this out: `correction = ||r|| * sqrt(pi/2) / m * <S @ q, signs>`. With `S` having i.i.d. N(0,1) entries (no `1/sqrt(m)` normalization in `generate_qjl_matrix`), the unbiased estimator is `(sqrt(pi/2)/m) * <Sq, sign(Sr)> * ||r||` only when the variance bookkeeping is paired with `S` being unnormalized — let me verify... Actually checking the paper (eq 5): `<x, y> ≈ (sqrt(pi/2)/m) * sum_i <Sq, sign(Sr)>_i * ||r||` where `S ~ N(0, 1)` (unnormalized). So the formula is correct **only for `m = d`**, which the default uses. But if anyone passes `qjl_dim != d` (which the code allows), the constant will be off. Since this code only ships with `m = d = head_dim`, the default works, but **flag for production**: hard-code `m = d` or fail at init if `m != d`.

- `turboquantdc/estimator.py:190` — **`(term1 + term2) * vec_norm` applies the *key* norm to a `(Q, K)` matrix, broadcasting across queries.** Given `vec_norm` shape `(K,)`, the broadcast onto `(Q, K)` is correct (numpy/PyTorch broadcasts across leftmost dims). But there is no analogous query norm handling — the formula assumes `query` is **already** in the same scale as `k_normalized * ||k||`. Since vLLM passes raw queries (post-Wq projection, post-RoPE, fp16, **not normalized**), and the estimator multiplies by `||k||` only, the result is `<q, k_unit> * ||k|| ≈ <q, k>` — which is correct. **But there is no test or assertion on this contract**, and a future change to also normalize queries (e.g. for a different scoring mode) would silently double-normalize. Add a docstring assertion: "queries are NOT normalized internally; norms are baked into vec_norm of the cached key only."

- `turboquantdc/polarquant.py:67-69,82-84` — **`Pi` is materialized as a dense d×d buffer even in WHT mode**, defeating the O(d) memory advantage. The comment "Build explicit Pi for API compatibility" is honest but the `register_buffer("Pi", ...)` will balloon GPU memory by `num_layers * d^2 * 4 bytes` (e.g. 64 * 128 * 128 * 4 = 4 MB per backend instance — small in absolute terms, but per-layer × per-rank blows up when sharded). **Fix:** make Pi computed on demand via `@property`, not stored. Not a correctness bug but it lies about the savings.

- `turboquantdc/vllm_integration.py:328` — **`output.view(*original_shape[:-1], self.head_dim)`** will fail with `RuntimeError: view size is not compatible` when `attn_weights @ v_reconstructed` produces a non-contiguous output (which it can when `Q != original_shape[0] * ... * original_shape[-2]`). For example, `queries` shape `(B, H, T, D)` flattens to `(B*H*T, D)`, returns `(B*H*T, D)` after attention, and `view(B, H, T, D)` should work — but only if the matmul output is contiguous. PyTorch's matmul almost always returns contiguous results, so this is *probably* fine, but the bigger issue: **the reshape `original_shape[:-1] + (head_dim,)` assumes the leading dims are preserved**, which is only true if no sequence-position rearrangement happened in the K dimension. For a **decode step** with `queries: (B, H, 1, D)` and cached `K: (K_total,)`, the output is `(B*H*1, D)` which `view(B, H, 1, D)` recovers fine. **For prefill** with `queries: (B, H, T, D)` against `K: (K_total,)` (where the cache contains all T keys), the result is `(B*H*T, D)` from the matmul... but only if the K dimension of `attn_weights` is `K_total = T` (single-batch prefill). With **batched prefill** (multiple sequences), `K_total != T` per sequence, the queries from sequence A would attend to keys from sequence B. The flat `(N, head_dim)` reshape **destroys batch boundaries**. **Fix:** the backend must not flatten `batch` and `seq` together; it must process per-sequence. As a stopgap, document that this only supports `batch_size=1`.

- `turboquantdc/kv_cache.py:78-107` — **`append` stores per-call dicts in a Python list** (`self._key_store: List[Dict]`). At seq_len=32768 with 64 layers, that's 32768 × 64 = ~2M Python dicts. Each `_gather_keys` call (line 227-256, called once per attention step) does `torch.cat` over **all** of them — O(seq_len) Python loop on every decode step. At decode-time aggregate of 4500 tok/s, that's 4500 × 64 layers × 2M-element cat = **catastrophic CPU bottleneck**. The class is documented as "drop-in" but is unusable for vLLM-scale workloads. **Fix:** preallocate flat tensors, write at index, use `[:seq_len]` slicing instead of cat.

- `turboquantdc/kv_cache.py:107` — **`squeeze()` without `dim=` argument** silently collapses any size-1 dimension. If `value_norms` happens to have shape `(1,)` (single token, single head), `value_norms.squeeze()` returns a 0-d scalar, breaking the downstream `unsqueeze(-1)` math at line 162. **Fix:** use `squeeze(0)` or `squeeze(-1)` explicitly.

- `turboquantdc/vllm_integration.py:570-620` — **`store()` calls `.float()` and re-runs Lloyd-Max quantization on every single decode step** (line 590: `self._key_quantizers[layer_idx].quantize(k_flat)`). The quantizer's internal `polar.quantize` does a brute-force argmin over `n_levels` centroids per coordinate per token (codebook.py:211). For 3-bit (8 levels) × 128 dims × 8 heads × 1 token × 64 layers = 65 K argmin ops per token — fine. But each one allocates an `(N, D, n_levels)` tensor for the broadcast subtract, then argmin. That's 8 × 128 × 8 × 4 bytes = 32 KB per layer per token. Cumulatively negligible, but the kernel launch overhead is ~5 μs × 64 layers × 4500 tok/s = 1.4 seconds/sec just in launch overhead. **Fix:** fuse the quantize into a single CUDA kernel (mentioned in repo as `cuda_kernels.py` — verify it's actually wired in here, it does not appear to be).

- `turboquantdc/vllm_integration.py:53` — **monkey-patch closure bug.** The example code defines `original_forward = attn_layer.self_attn.attn.forward` outside the inner `make_patched_forward` and never uses it. Worse, the `patched_forward` calls `backend.compress_kv` and `backend.compute_attention` but **completely bypasses** the original attention. This is fine only if the backend is a complete reimplementation, which it currently is *not* (no causal masking by default, no GQA broadcasting, no scale alignment with vLLM's softmax dtype). The example will run but produce incorrect outputs vs the reference. **Fix:** either flesh out the example or remove it and replace with a clearly-marked TODO.

- `turboquantdc/vllm_integration.py:300-329` — **GQA (`num_kv_heads != num_heads`) is not handled.** The flatten `_flatten_to_2d(queries)` collapses `(B, num_heads, T, D)` to `(B*num_heads*T, D)`. Keys come in as `(B, num_kv_heads, T, D)` flattened to `(B*num_kv_heads*T, D)`. Then `query @ x_mse.T` (estimator.py:179) computes a `(B*num_heads*T, B*num_kv_heads*T)` score matrix, which does **not** correspond to "each query head attends to its mapped KV head". The result is a full cross product. For Qwen3-7B (num_heads=28, num_kv_heads=4), this gives a **7x denser score matrix and incorrect attention**. **Fix:** the backend must take `num_heads` as well, repeat-interleave keys across the GQA group factor, *or* compute scores per-head-group. Given the 6-hour ship target on Qwen3.6-27B (`num_kv_heads=8` per the lookup at line 795), this is a **must-fix**.

- `turboquantdc/vllm_integration.py:322` — **`v_reconstructed * v_norms.unsqueeze(-1)` assumes `v_norms` is 1-D**, but on the CacheManager path it returns `(K*H,)` (line 681) which is 1-D after `reshape(K * H)`. Fine. But on the AttentionBackend path, `compressed_values = (v_indices, v_norm.squeeze(-1))` (line 256) where `v_norm` started as `(N, 1)` so squeeze yields `(N,)` — also 1-D. Both OK. **However** `v_reconstructed = self._value_quantizers[layer_idx].dequantize(v_indices)` returns shape `(K*H, D)`; the `unsqueeze(-1)` makes `v_norms` `(K*H, 1)`, broadcast onto `(K*H, D)`. Correct **only if** the indexing is "flatten first then dequant" — which it is. But the `attn_weights @ v_reconstructed` then has `attn_weights: (Q*H_q*T_q, K*H_kv*T_kv)` and `v_reconstructed: (K*H_kv*T_kv, D)` — same GQA bug as above; queries attending across all KV heads.

## MEDIUM  (correctness gaps that probably won't bite the demo but should be tracked)

- `turboquantdc/estimator.py:99` — **`vec_norm + 1e-8` is fp32 (input dtype-dependent) but the divisor for fp16 inputs would round to `vec_norm` directly**. Already covered as fp16 underflow above; tracked here as a hardening item: use `clamp_min(eps)` not `+ eps`.

- `turboquantdc/qjl.py:76` — **`residual @ self.S.T` on every key compression**. With `S` shape `(m=d, d) = (128, 128)`, `S.T` is recomputed on each call (cheap, but if `S` is non-contiguous after `.to(device)`, this is a strided read). **Fix:** cache `self.S_T = self.S.t().contiguous()` at init.

- `turboquantdc/qjl.py:106` — same comment, query side.

- `turboquantdc/polarquant.py:144` — **`self.codebook.quantize(y)`** uses `LloydMaxCodebook.quantize` which does `centroids = self.centroids.to(x.device)` on every call (codebook.py:209). For fp16 vs fp32 mismatch (centroids are fp32, queries may be fp16), the `.to(device)` is a no-op but the subtract `x.unsqueeze(-1) - centroids` will silently up-cast `x` to fp32. That's correct but allocates a `(N, D, n_levels)` fp32 tensor every call — non-trivial garbage on the hot path.

- `turboquantdc/kv_cache.py:121-124` — **`device=queries.device`** vs `self.device`: if the user constructed the cache on CPU but passes CUDA queries, the empty-cache fast path returns a CUDA tensor while `_gather_keys()` would return CPU tensors. Inconsistent.

- `turboquantdc/vllm_integration.py:171-179` — **per-layer `seed=layer_idx * 1000`** at layer 0 yields `seed=0`. PyTorch's `manual_seed(0)` is valid but a common "this looks unset" value. Slightly safer to start at `seed_base + layer_idx`.

- `turboquantdc/vllm_integration.py:887` — **`estimated_max_context = int(cache_budget_bytes / bytes_per_token)`** uses 40% of VRAM as cache budget. For a 24 GB 4090 running Qwen3.6-27B (which is ~14 GB at 4-bit weights, ~27 GB at 8-bit), the 60% reserved for weights does not actually fit the model weights. The function does not validate this — it returns a "max context" that assumes the weights fit. Add a `weight_bytes` parameter or a sanity check.

- `turboquantdc/estimator.py:192-193` — **`if squeeze_q and result.dim() > 0 and result.shape[0] == 1: result = result.squeeze(0)`** — this collapses based on heuristic, not based on the original input rank. If the user passes `query: (1, d)` (i.e. *intentionally* batch=1), the result will be silently collapsed. The `squeeze_q` flag was set only when `query.dim() == 1`, so this branch is gated correctly — but the additional `result.shape[0] == 1` check is dead unless the result is from broadcasting against a 1-key cache. Confusing logic; refactor to track squeeze intent only on the input.

- `turboquantdc/vllm_integration.py:597` — **`((ck["qjl_signs"] + 1.0) * 0.5).to(torch.int8)`** is correct for `{-1, +1} → {0, 1}` but fails silently for any sign value not exactly ±1. `qjl.project_and_sign` uses `(projected >= 0).float() * 2.0 - 1.0`, which is exactly ±1, so safe. Add `assert` to guard against future refactors.

- `turboquantdc/qjl.py:32-79` — `self.S` is a dense fp32 `(m, d)` matrix kept on GPU permanently. With 64 layers × 128² × 4 bytes = 4 MB. Fine. But for **multi-rank tensor-parallel** vLLM serving, this gets duplicated across ranks. Not a correctness bug but a pattern to flag.

- `turboquantdc/vllm_integration.py:597-599` — **packing QJL signs to int8** wastes 7 bits per sign (paper says 1 bit, code uses 8). The docstring at line 422 says "int8 (packed ±1 as {0,1})" — but this isn't packed, it's stored 1-per-byte. With `S=32768, H=8, M=128`, that's 32 MB per layer, 2 GB across 64 layers — **8x more than the paper claims**. **Fix:** use `torch.bitwise` packing or `torch.uint8` with bit-shift logic (this may also need a CUDA kernel for the dot product on packed signs).

- `turboquantdc/vllm_integration.py:660-666` — **`buf[slot_indices]` is a fancy-indexing gather** — non-contiguous output, allocates a new tensor each call. With seq_len=32768 and 4500 tok/s, that's 4500 × 64 layers = 288 K gathers/sec, each allocating ~1 MB. Pressure on the CUDA caching allocator.

## Summary recommendation

**Block.** Out of the seven CRITICAL findings, three (no-state in `TurboQuantAttentionBackend`, GQA broken in `compute_attention`, fp16 underflow in `quantize`) will prevent any correct Qwen3-style autoregressive decoding **at all**. Two more (paged-block layout, fp32 indices) prevent reaching the 16-32 concurrent-stream throughput target. The remaining two (fast_wht in-place mutation, PolarQuant unit-vector contract) are landmines that will fire during the first non-toy run.

For the 6-hour ship window, the minimum-viable fix list is:

1. Make `TurboQuantAttentionBackend` stateful: store compressed K/V per (layer, sequence) and let `compute_attention` see history.
2. Add GQA broadcasting: repeat-interleave compressed keys by `num_heads / num_kv_heads` before the score matmul, **not** flatten heads into batch.
3. Cast queries and norms to fp32 inside the estimator (or at least clamp_min the divisor to 1/65504).
4. Replace the Python-list KV store with a preallocated tensor or accept that `kv_cache.py` is a reference impl, not the production path.
5. Document explicitly: "no PagedAttention support, single-sequence demo only" — and keep the cache manager out of the demo.
