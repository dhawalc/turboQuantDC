Verification of the `29x speedup at d=256, register cliff` claim
---------------------------------------------------------------
The number is plausible but the *cause* is mis-described in `HANDOFF.md` and `benchmarks/gemma4_showcase.py:251,455`. The Triton dequantize kernel at `triton_kernels.py:208-256` materializes the full `R` matrix as a `(BLOCK_D, BLOCK_D)` tile in registers (`r_block` at line 239). At d=128, `BLOCK_D=128` -> 16 K floats = 64 KB tile, which is already huge but Triton spills to local memory. At d=256 the tile is 256*256*4 = 256 KB, which absolutely cannot live in registers (Ada has 64 KB regfile per SM total) and is forced into local memory (DRAM-backed L1). That is a **spill-to-local cliff**, not a "register cliff" in the SM-89 architectural sense.

Meanwhile the CUDA kernel (`dequantize.cu:78-86`) streams `R` directly from global memory in the inner loop without ever building a tile. So the speedup is not "the CUDA kernel is fast", it is "the Triton kernel falls off a cliff because of how its tile was written". Replacing the Triton kernel with a streaming version would close most of the gap. Treat the 29x as a Triton-baseline artifact, not a portable performance claim, and remove the misleading wording from the public showcase.

I cannot rerun the benchmark from this read-only review, but the existing benchmark in `benchmarks/bench_cuda_kernels.py:115-164` only covers d=128, 256 with random R/centroids. There is no measurement against a *good* baseline (cuBLAS gather + matmul, or FlashInfer's quant kernels). The PyTorch reference is `centroids[indices.long()] @ R * vec_norm` which delegates to a 1-token cuBLAS gemm and is genuinely slow at small batch — the speedup vs PyTorch is real but not the metric people care about.

## CRITICAL  (kernel produces wrong output / crashes / unsafe memory access)
- `turboquantdc/cuda/wht.cu:84-95` — `wht_shared_kernel` has a correctness hazard for `D=64, NTHREADS=32`. The butterfly stage updates `s_v[lo]` and `s_v[hi]` in-place. `__syncthreads()` is called after the for-loop completes (line 94). However, when `D >= 2*NTHREADS` (e.g. D=2048, NTHREADS=256, 1024 pairs / 256 threads = 4 pairs/thread), the `for (int p = tid; p < N_PAIRS; p += NTHREADS)` loop has each thread touch four disjoint pair offsets. Within one stage, two different pairs handled by the same thread can target the SAME `lo` or `hi` if the geometry collides — which actually does NOT happen here because pair indices are unique and the lo/hi mapping `(group*2*h + element, lo+h)` is a bijection with [0, D). So no within-thread RAW. **However**, between stages, you need `__syncthreads()` to make stage `h` visible to all threads before stage `2h` reads. That is present (line 94). Also need a sync between the optional sign-flip block (lines 60-65) and the first butterfly read; that is present (line 64). I traced the data flow and **the WHT kernel is functionally correct**. Downgraded to LOW; left here only to record the audit.
- `turboquantdc/cuda/dequantize.cu:78-86, 138-146` — silent overflow when `MAX_CENTROIDS` is exceeded by future configs. The TORCH_CHECK at line 270 / 332 enforces `n_cent <= 16` (4-bit), but `residual_quant.py` plus E8 codec experiments may push to 256-entry codebooks. When that happens the kernel will refuse to launch — fine — but `cuda_dequantize` in `cuda_kernels.py:110-116` does not catch this and will throw a hard RuntimeError into the model forward, killing generation mid-sequence. Gate the call in Python and fall through to the PyTorch path when `centroids.numel() > 16`.
- `turboquantdc/cuda/dequantize.cu:181-184, 231-233` — generic kernel reads `s_centroids[idx]` with `idx` loaded from global memory (`row_indices[i]`). There is **no bounds check on `idx`** inside the kernel. If a corrupted index ever lands (e.g. uninitialized memory, a broken upstream quantize, or vLLM passes garbage in a paged block), the kernel will out-of-bounds-read `s_y_hat[i] = s_centroids[idx]` and either silently return junk or trip a CUDA illegal-memory-access asynchronously, killing the whole stream. Add `idx = min(idx, n_centroids - 1)` or assert at the device level under DEBUG, and at minimum run a sanity check in the Python wrapper (`indices.max() < n_centroids`) when `TURBOQUANTDC_VALIDATE=1`.
- `turboquantdc/cuda_kernels.py:111-116` — `_CUDA_DEQUANTIZE.dequantize_mse(...)` is called with tensors that are forced to `int32`/`float32`. The kernel signature requires `int32`, but `indices.contiguous().to(torch.int32)` loses information silently if the upstream produced int64 indices > 2^31 (won't happen for codebooks of size 16, but the conversion masks bugs). The bigger issue is **dtype incompatibility with HF/vLLM's bf16 path**: `R.contiguous().float()` materializes a fresh fp32 copy of the rotation matrix on every call. For a 256x256 matrix that is 256 KB allocated and copied per attention step. Cache the float copy on the layer.
- `turboquantdc/cuda_kernels.py:208, 247` — `_CUDA_WHT.wht(x_flat, signs.contiguous().float(), False)` returns a tensor that is then `result.reshape(orig_shape)` where `orig_shape` may have been bf16/fp16 originally. The Triton/CUDA path silently upcasts to fp32 and *never restores the original dtype*. Downstream attention expects bf16 matmuls. This will both blow VRAM and crash GQA repeat_interleave at the call site if dtypes diverge. Cast back to `x.dtype` before reshape.

## HIGH  (clear performance gain on 4090 / production blocker)
- `turboquantdc/cuda/dequantize.cu:78-86` — the inner-rotation loop reads `R[k * D + j]` from global memory for every `(j, k)` pair. With THREADS_PER_BLOCK=128 and D=128, each thread does 128 * 4 B = 512 B of global loads from R per output element it owns; **the same `R[k][j]` is reread by 128 different blocks (one per vector)**. R is at most 256 KB (256*256*4) and is reused for the entire batch and for every attention step. Stage R in shared memory once per block (or better, into constant memory if it fits in 64 KB), or use `__ldg()`/the read-only cache. Right now this kernel is bandwidth-bound on a value that should hit L1/L2 100% but isn't being hinted as read-only. Adding `__ldg(&R[k * D + j])` is a 1-line change that should give 1.5-2x on Ada because the L1 read-only cache becomes effective.
- `turboquantdc/cuda/dequantize.cu:78-86` — `R[k * D + j]` for fixed `j` strides by D between successive `k`. This is a **column-major access pattern with stride D over a row-major matrix**, which is the worst case: at D=256 each load crosses cache lines, and threads in a warp all reading the same `k` (different `j`) DO coalesce, but reads across the inner `k` loop drag in 128 different cache lines. Transpose R once on the host so the kernel can do a row-major sweep, or reorganize so threads cooperate on a tiled GEMV. Without this fix the d=256 path is leaving a large factor on the table — very visible in the `29x vs Triton` baseline that actually came from Triton being even worse.
- `turboquantdc/cuda/dequantize.cu:38-87, 96-147` — `THREADS_PER_BLOCK=128` (line 24) means at D=128 each thread does the entire `k=0..127` reduction sequentially. With 128 threads per output element being 1:1 with output dims, you get **no warp-level cooperation on the reduction**. On Ada, `__shfl_xor_sync` warp reductions or split-K reductions would let two warps collaborate per output and roughly halve latency. For the d=256 path each thread already does 2 outputs, which is fine, but the reduction is still purely sequential per thread. Cooperative warp reduction is a clean 1.5x.
- `turboquantdc/cuda/dequantize.cu:62, 120` — `__shared__ float s_y_hat[D]` is allocated even though it is read once per output element in the inner loop. Each thread reads `s_y_hat[k]` for k=0..D-1 and `R[k*D+j]`. With 128 threads in a warp configuration, `s_y_hat[k]` is broadcast across the warp (all threads read the same `k`) -> bank-conflict-free (broadcast). But the alternative — keeping `y_hat` per-thread in registers via a small per-thread tile and using `__shfl` to share — would skip shared memory entirely. For D=128 keeping the whole vector in 8 registers/thread is feasible. This is a candidate for a register-resident kernel; estimated 1.3-1.6x.
- `turboquantdc/cuda/dequantize.cu:38, 97, 155, 200` — the kernels are templated for D=128 and D=256 only. **vLLM PagedAttention typically uses head_dim=128 (Llama, Qwen, Mistral) or 64 (some 2B-class), and 256 is rare** (only Phi-3 14B and certain MoEs). The d=64 path falls to the generic kernel which has no `#pragma unroll` (lines 192). Add a `D=64` template specialization; this is the more common production case than D=256.
- `turboquantdc/cuda/dequantize.cu` (entire file) — uses `float` (fp32) input and output throughout. The HF / vLLM forward pass runs in bf16 or fp16. Every call therefore pays an upcast on input (`vec_norms`, `centroids`, `R`) and downcast on output. The kernel **does not support bf16 / fp16 directly**, which means there is a `.float()` materialization in the Python wrapper (`cuda_kernels.py:112-115, 156-161`) that doubles the bytes of every `R` move. Add bf16 templates: the inner reduction can stay in fp32 (Ada has fast bf16->fp32 mma instructions), but inputs and outputs should be bf16 to avoid the format-conversion tax.
- `turboquantdc/cuda/dequantize.cu:24` — `THREADS_PER_BLOCK = 128` is used for both d=128 and d=256. For d=256 you only get 50% occupancy on the matvec because each thread does 2 outputs, but the inner reduction is still 256 muladds. Bumping to 256 threads/block for d=256 is an obvious tune. Without this, an Ada SM with 1536 active threads/SM caps at 6 active blocks; with 256 threads it is 6 blocks of 256 = 1536, exactly maxing the warp scheduler. Currently the 128-thread case under-utilizes.
- `turboquantdc/cuda/dequantize.cu:155-247` — generic kernels reuse `THREADS_PER_BLOCK=128` regardless of D. For D=512+ this is severely under-threaded and the `for i = tid; i < D; i += THREADS_PER_BLOCK` loops hit their iteration limit. Either dispatch threads = min(1024, next_pow2(d)) or template more aggressively.
- `turboquantdc/cuda/wht.cu:60-64, 98-110` — for the forward path the sign multiplication is fused (good), but the inverse path multiplies by signs *after the WHT* in the store loop (lines 102-104). The ordering matches `apply_wht_rotation` semantics in `rotation.py:205-211`, so it is correct. But the kernel does **two passes over `s_v`** (load, then sign, then butterfly, then store-with-sign), each guarded by a `__syncthreads`. The sign step at lines 60-64 could be folded into the load (lines 53-55) at zero cost and saves a sync.
- `turboquantdc/cuda/wht.cu:84-93` — the integer divide `int group = p / h;` is computed on every iteration of every stage. The comment claims power-of-2 strength reduction kicks in (line 78), but `h` is a runtime variable in the outer for-loop, so the compiler **cannot** apply the optimization. Use `__ffs(h) - 1` once per stage as a shift count and replace `p / h` with `p >> shift` and `p % h` with `p & (h-1)`. This is hot-path arithmetic in every butterfly stage; expect 1.2-1.3x on the WHT.
- `turboquantdc/cuda/build.py:43-51` — `--use_fast_math` is on globally. For a quantization kernel that is fine for the matmuls, but it also changes `1.0f / sqrtf(D)` (`wht.cu:98`) into `rsqrtf` *and* enables flush-to-zero. The WHT division `1/sqrt(d)` is a constant — precompute it on the host once and pass it as a kernel argument or as `constexpr` instead of letting the kernel call `sqrtf` per-vector.
- `turboquantdc/cuda/build.py:29-40` — the arch detection picks the running GPU's SM, but `cap = torch.cuda.get_device_capability(0)` always uses device 0. In a multi-GPU vLLM deployment with mixed GPUs (rare, but possible: 4090 + 3090) this misbuilds. Use `torch.cuda.current_device()`. Also missing: `--threads 4` to parallelize nvcc, `-Xptxas -v` is off so register usage is invisible, and there is **no `--ptxas-options=-O3`** explicitly. Add `-lineinfo` for Nsight Compute profiling.
- `turboquantdc/cuda/build.py:71-77, 103-109` — `verbose=False` hides PTXAS register-use output. There is no way to verify the "register cliff" claim from build artifacts. Set verbose during the development build, or save the PTXAS log to `_CACHE_DIR/build.log`.
- `turboquantdc/triton_kernels.py:208-256, 264-313` — both Triton dequantize kernels load the full `(BLOCK_D, BLOCK_D)` rotation tile (lines 239-243, 302-306). At d=256 this is 256 KB of *register-or-spill* tile. PTX will spill to local memory aggressively. The fix is to tile the inner-product loop along K with `BLOCK_K=32` and accumulate, never materializing the full R tile. This is the actual cause of the 29x gap and a 1-day fix.
- `turboquantdc/triton_kernels.py:359, 414, 450, 489` — no `num_warps` or `num_stages` are specified on any `@triton.jit` launch. Triton defaults are `num_warps=4, num_stages=2-3` which is suboptimal for memory-bound dequantize. For an Ada GEMV-like kernel `num_warps=8, num_stages=4` is the typical sweet spot. Run a small autotuner.
- `turboquantdc/triton_kernels.py:175-199` — `_inner_product_kernel` has `BLOCK_K=16` keys per program with a serial `for ki in range(BLOCK_K)` *inside* the kernel. This serializes 16 keys per program, no parallelism across them. Replace with a 2D `tl.arange` over keys so the gather and dot become tile operations. As written this kernel can never beat cuBLAS for any meaningful key count and the comment claiming "fusion beats pure PyTorch" (lines 124-130) is unverified — there is no benchmark of `triton_inner_product` in `bench_cuda_kernels.py`. The IP estimator never made it into the CUDA path (`cuda_kernels.py:488-536` falls back to Triton). For vLLM this is the hot path; consider it production-blocking.
- `turboquantdc/fused_attention.py:93, 98, 107, 111, 119` — `query_states.float()`, `centroids.float()`, `qjl_matrix.float().T`, and `mse_indices.long()` all create fp32/int64 tensors on every call. In a decode loop with 80 layers, this is 80 spurious allocations per token. Pre-cast and cache these on the cache object once.
- `turboquantdc/fused_attention.py:98` — `centroids.float()[mse_indices.long()]` is a fp32 gather with shape `(batch, n_kv_heads, seq_kv, d)`. At seq_kv = 32K, batch=1, heads=8, d=128 this is 32K * 8 * 128 * 4 B = **128 MB of materialization on every decode step**. This defeats the whole point of compressed KV. The fused kernel should gather *into* the matmul, not materialize a fp32 dense tensor first. This is a vLLM-readiness blocker — a 32K-token decode at 80 layers will allocate 10 GB of scratch. Use `torch.compile`'s gather fusion at minimum, or write a Triton kernel that does the gather + matmul together.
- `turboquantdc/fused_attention.py:147, 178, 220` — `value_states.repeat_interleave(heads_per_kv, dim=1)` materializes the GQA expansion. Same blow-up: 32K * 32 * 128 * 2 B = 256 MB at common GQA ratios. vLLM's PagedAttention specifically avoids this; integrating with it requires *not* expanding values and instead broadcasting in the matmul.
- `turboquantdc/cuda/dequantize.cu`, `wht.cu` — **no Tensor Core usage anywhere**. The reduction in dequantize is a `(1, D) @ (D, D)` matvec — at batch=1 no TC, but with batch > 1 (which IS the case in vLLM prefill where you decompress many tokens at once) you have a `(N, D) @ (D, D)` GEMM that should hit `wmma.m16n16k16` bf16 tensor cores on SM 89. Instead the kernel launches one block *per vector*. For prefill of 32K tokens at d=128, you launch 32K blocks of 128 threads doing scalar fma; cuBLAS would do this in ~5 us. **Add a batched GEMM path that calls `cublasGemmStridedBatchedEx` (or just `at::matmul`) when batch >= 8** and only use the per-vector kernel for the decode-1 case.
- `turboquantdc/fused_attention.py` (whole file) — the kernel name is misleading. There is **no fused CUDA/Triton kernel** here; this is pure PyTorch eager code with seven separate `matmul`/`repeat_interleave`/`softmax` ops. For 6-hour vLLM ship goal, decide whether to (a) actually fuse this in Triton (multi-day) or (b) accept eager mode and document it. Right now the name promises performance the file doesn't deliver.

## MEDIUM  (good optimization to track but not blocker)
- `turboquantdc/cuda/dequantize.cu:53, 114` — `__shared__ float s_centroids[MAX_CENTROIDS]` is 64 B = 16 banks worth on a 32-bank shared memory. With broadcast access (all threads read same `idx[k]`) there are no conflicts. Fine, but if `MAX_CENTROIDS` ever grows to 64 (6-bit codebook) the access pattern needs review. Padding to `MAX_CENTROIDS + 1` would make it conflict-resilient generically.
- `turboquantdc/cuda/dequantize.cu:62, 120, 170-172, 217-219` — `s_y_hat[D]` shared memory. At D=256 that is 1024 B; at D=512 with the generic path it's 2 KB. With `MAX_CENTROIDS + D` shared per block, occupancy is fine on Ada (128 KB shared/SM), but for very large D (1024) the generic kernel reserves 4 KB and on top of `s_centroids` may cause scheduler limits to kick in. Compute the smem footprint and clamp `THREADS_PER_BLOCK` accordingly.
- `turboquantdc/cuda/dequantize.cu:65, 78, 127, 138` — `#pragma unroll` (no count) followed by `#pragma unroll 8` is inconsistent. `#pragma unroll` with no number requests *full* unroll. For the outer `for (j = tid; j < D; j += 128)` loop with D=256 that's 2 iterations, fine. But for `for (k = 0; k < D; k++)` with `#pragma unroll 8`, full unrolling at D=128 = 16 inner units, which can balloon the register usage. Profile register count with `-Xptxas -v` (currently disabled, see HIGH on build.py).
- `turboquantdc/cuda/dequantize.cu:115-125` — `dequantize_residual_kernel` does the `__syncthreads()` (line 125) AFTER reading `scale = res_scale[vec_id]` (line 123) but BEFORE the y_hat fill loop. Fine, but the sync placement is asymmetric vs the MSE kernel (which syncs at line 57). Consistency improves auditability.
- `turboquantdc/cuda/wht.cu:48` — `__shared__ float s_v[D]` for D=2048 is 8 KB. With NTHREADS=256 and 8 KB shared, each SM can host (128 KB / 8 KB) = 16 blocks but only (1536 / 256) = 6 warp budget, so 6 blocks → 6 vectors per SM. The current parameterization is OK but undocumented.
- `turboquantdc/cuda/wht.cu:144-176` — the dispatch is hand-rolled for 6 specific D values. Missing D=384 and D=192 (Phi-3) — they fall into the `default: TORCH_CHECK(false, ...)` branch and crash. The Triton path supports any power of 2; CUDA does not. If a model with d=64*3 or 96 hits this, you crash.
- `turboquantdc/cuda/wht.cu:98` — `1.0f / sqrtf((float)D)` is computed in every thread of every block. Hoist to a kernel argument or use `rsqrtf`.
- `turboquantdc/cuda/wht.cu:188-191` — `py::arg("inverse") = false` default uses `bool` but the kernel takes `int inverse`. Works because of implicit conversion but is fragile. Define an explicit enum.
- `turboquantdc/cuda_kernels.py:45-68` — `_ensure_backend()` uses module-level globals for caching but doesn't lock. The comment says "Thread-safe via Python GIL" (line 46), which is true for pure Python state. However, the JIT compilation (`load(...)`) is *not* GIL-protected and races on the cache directory if two processes start simultaneously (e.g., vLLM tensor-parallel workers). Add a file lock around `_CACHE_DIR`.
- `turboquantdc/cuda_kernels.py:122-126, 168-176, 219-220, 257-259` — fallback chain catches `ImportError` and `RuntimeError`. A Triton kernel that hits a CUDA OOM raises `torch.cuda.OutOfMemoryError`, which is a `RuntimeError` subclass — caught here, but then the PyTorch fallback also OOMs. Surface OOM specifically rather than silently degrading.
- `turboquantdc/cuda_kernels.py:373` — `self.SR = (self.S @ self.R.T).contiguous()` allocates a (m, d) fp32 tensor. For m=d=128 that is 64 KB, fine. For d=256 it is 256 KB per layer. With 80 layers and 32 KV heads, this is ~640 MB of materialized SR. Recompute on demand or share across layers with the same R/S.
- `turboquantdc/cuda_kernels.py:412` — `(x / (vec_norm + 1e-8))` uses `1e-8` regularization. For bf16 this rounds to ~0 and provides no numerical safety. Use `1e-4` or `torch.finfo(x.dtype).eps`.
- `turboquantdc/triton_kernels.py:817` — `d_float = d + 0.0` to "promote constexpr int to float" is a Triton 2.x idiom. Triton 3.x supports `tl.cast(d, tl.float32)`. Update for Triton 3.6 (the file header claims this is the target).
- `turboquantdc/triton_kernels.py:738-815` — the WHT butterfly is unrolled by hand for stages 1..256 (log_d > 0..8). For d=2048 you would need stage 512 (log_d > 9), which is missing. Triton path is therefore capped at d=512 (matches the wrapper claim at `cuda_kernels.py:212`). Document the cap clearly and consider auto-generating the stages with a Python codegen step.
- `turboquantdc/fused_attention.py:174` — `weights = F.softmax(scores, dim=-1, dtype=torch.float32).to(dtype)` casts back to bf16 before `weights @ v`. The cast is correct but the upcast-softmax-downcast pattern allocates a fp32 weights tensor of size `(B, H, Q, K)` — at K=32K, B=1, H=32, Q=1 that is 4 MB per layer; manageable but not free. Consider FlashAttention-style fused softmax.
- `turboquantdc/fused_attention.py:147, 200, 220` — `value_states.repeat_interleave(...)` is called on every layer. At GQA ratio 4 and seq=32K, this is 32K * 32 * 128 * 2 B = 256 MB allocated and freed every step. Reuse a buffer.
- `turboquantdc/custom_attention.py:88-103` — flatten + dequantize + reshape pattern allocates two intermediate tensors. Same buffer-reuse opportunity.
- `turboquantdc/custom_attention.py:395-407` — `apply_rotary_pos_emb` import dance is fragile. If transformers updates the module path, generation fails silently. Add a smoke test.
- `turboquantdc/cuda/dequantize.cu:269, 332` — error messages for shape mismatch are concise but not helpful. Include the actual sizes received: `TORCH_CHECK(R.size(0) == d, "R rows = ", R.size(0), " expected ", d)`.

## vLLM PagedAttention compatibility
Specific blockers for shipping into vLLM (which the goal mandates within 6 hours):

1. **Per-vector launch granularity** (`dequantize.cu:47, 108`): one block per vector. PagedAttention uses 16-token blocks contiguous in memory. If your kernel launches 16 blocks per page when 1 block per page (handling 16 vectors with 16-thread-warp cooperation) would suffice, you waste launch overhead. Refactor to take `(num_blocks, block_size=16)` and have each CUDA block process one page.
2. **No paged layout support**: indices in dequantize.cu are assumed contiguous `(batch, d)`. vLLM passes a `block_table` of int32 page IDs and expects the kernel to gather. Add a `block_table`-aware dispatch.
3. **fp32 I/O** as discussed in HIGH; vLLM's KV cache is bf16/fp16.
4. **No CUDA stream argument**: every kernel launches on the default stream. vLLM uses non-default streams for TP/PP overlap. Accept a `c10::cuda::CUDAStream`.
5. **Quantize kernel**: `cuda_kernels.py:415-419` falls back to Triton for quantize. vLLM's prefill compresses thousands of tokens at once, so quantize is also hot. Port `_quantize_kernel` to CUDA or at least to a tiled Triton kernel.
6. **GQA broadcast in fused_attention.py**: `repeat_interleave` is materialization. vLLM avoids this entirely.
7. **No support for d=64**: the templated CUDA kernels are 128/256 only. Many small GQA models (Qwen3-1.7B-A350M variants) use d=64. Falls to generic kernel which is significantly slower.

## Summary of decision
**BLOCK** ship-to-vLLM-in-6-hours.

Critical and HIGH issues:
- Silent fp32 materialization (HIGH) wipes out the compression benefit at long context.
- No paged-layout support (vLLM blocker).
- No bf16 input dtype (vLLM blocker).
- Inner-product kernel never ported to CUDA (vLLM hot path).
- `fused_attention.py` is not actually fused.

The 29x speedup claim is **technically reproducible** but the reason given (`Triton register cliff`) misframes a Triton tile-size bug as an architectural fact. The CUDA kernel itself is leaving 2-4x on the table from the issues above (no `__ldg`, no R caching, suboptimal threads/block, no Tensor Cores, no bf16). A polished CUDA kernel would do 1-2x vs cuBLAS gemm baseline, not 29x vs a broken Triton baseline.

Recommend: 
1. Fix the CRITICAL items (illegal-index handling, dtype round-trip).
2. Add bf16 templates and `__ldg(R)`.
3. Port quantize and inner-product kernels to CUDA.
4. Add paged layout support.
5. Replace the eager `fused_attention.py` with an actual fused kernel.
Estimated 2-4 days of focused work, not 6 hours.
